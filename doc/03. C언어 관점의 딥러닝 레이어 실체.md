## C언어 관점의 딥러닝 레이어 실체

### 1. `nn.Embedding` = 메모리 선언 및 할당 (HashMap Lookup)

- 단어마다 고유의 8칸(혹은 512칸)짜리 배열 방을 할당해 두는 **딕셔너리 선언부**다.
    
- 여기서는 데이터가 섞이거나 변하는 연산이 없다. 문장이 들어오면 인덱스로 메모리 주소를 찾아 값을 쏙 꺼내오는(Lookup) 포인터 역할만 한다.
    

### 2. `nn.Linear` = 모든 연결 경우의 수의 집합 공간 (Matrix Transformation)

- 임베딩에서 꺼내온 데이터를 비틀고 섞어서 새로운 차원으로 변환하는 **실제 가공 공장**이다.
    
- `nn.Linear(8, 10)`을 선언하면 입력 8개와 출력 10개가 만나는 80개의 모든 연결 통로(포인터)가 개설된다.
    

### 3. C언어로 번역한 `nn.Linear` 구조체 레이아웃

실제 딥러닝 엔진 밑바닥을 까보면, 가중치(Weight)는 이중 배열을 1차원으로 편 플랫한 공간이고, 바이어스(Bias)는 구조체 선언부에 별도로 붙어 있는 포인터 배열이다.

```
typedef struct {
    int in_features;   // 8 (입력 차원)
    int out_features;  // 10 (출력 차원)
    
    float* weights;    // 크기 80짜리 연속된 float 1차원 배열 포인터 (가중치 이중 배열 역할)
    float* bias;       // 크기 10짜리 별도 float 배열 포인터 (출력방마다 더해지는 덤)
} LinearLayer;
```

- **왜 굳이 바이어스를 따로 뺄까?** 
  1. `bias=False` 옵션을 줬을 때 `layer->bias = NULL;` 처리만 하면 되므로 아키텍처가 깔끔해진다.
  2. GPU 가속기(CUDA/BLAS)가 순수 행렬 곱셈($A \times B$)을 가장 빠른 속도로 처리할 수 있도록 이중 배열 덩어리를 온전히 보존하기 위해서다. 곱셈이 끝난 버퍼 위에 바이어스 포인터를 돌며 덤으로 `+` 연산만 슥 가해준다.

### 4. 로우레벨 CUDA API의 동작 방식 (C/C++ 수준)

파이토치 내부에서는 `.to('cuda')`나 데이터의 `.cuda()` 호출을 감지하면 엔비디아의 드라이버 API를 호출하여 VRAM 주소 링크를 생성합니다.

```
// 1. GPU VRAM 영역에 80개짜리 float 연속 메모리 공간을 파라 (디바이스 포인터 뚫기)
float* d_weights; 
cudaMalloc((void**)&d_weights, sizeof(float) * 80);

// 2. 메인 RAM(Host)에 있던 가중치 데이터를 방금 개설한 GPU VRAM(Device) 주소로 복사(DMA 전송)
cudaMemcpy(d_weights, h_weights, sizeof(float) * 80, cudaMemcpyHostToDevice);
```

이 단계를 거치면 하드웨어 단에서 그래픽 카드가 다이렉트로 접근 가능한 하드웨어 통로 포인터(`d_weights`)가 활성화됩니다.

### 5. "포인터 던질 테니 찢어라" - CUDA 커널(Kernel) 호출

- 데이터 파이프라인을 타고 인풋 데이터가 쭉쭉 들어올 때, CPU는 연산을 직접 하지 않고 GPU 장치에게 포인터 주소만 던지며 명령을 하달합니다. 이를 **커널 호출**이라고 합니다.
    
- C언어에서 함수에 주소값을 아규먼트로 넘기는 것과 같지만, 하드웨어 명령어 지시문인 `<<<blocks, threads>>>`가 붙습니다.
    

```
// GPU 내의 수천 개 코어에게 입력, 가중치, 바이어스 포인터 주소를 던지며 일제히 연산 명령
linear_layer_kernel<<<blocks, threads>>>(d_inputs, d_weights, d_bias, d_outputs);
```

- 이 명령이 떨어지는 순간, GPU 내부에 대기하던 수천 개의 CUDA 코어들이 할당받은 주소 오프셋으로 일제히 다이렉트 접근하여 **수만 개의 행렬 곱셈 연산을 동시에(Parallel) 분할 처리**해 버립니다.
    
- CPU가 순차적인 `for` 루프 포인터 연산으로 쩔쩔맬 때, GPU는 하드웨어 통로에 고속 매핑된 주소들을 다이렉트로 긁어 연산하기 때문에 수만 배의 속도 차이가 발생하는 것입니다.

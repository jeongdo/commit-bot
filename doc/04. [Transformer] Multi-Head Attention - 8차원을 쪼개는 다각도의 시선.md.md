# Multi-Head Attention 개념 정리

---

## 1. Attention이 왜 필요한가

```
['버그', '수정'] → 각각 8차원 벡터로 변환
```

문제는 `수정`이 앞에 뭐가 오든 똑같은 벡터로 표현된다는 것.

```
버그 수정  →  수정 = [0.3, -0.1, 0.8, ...]   # fix 해야 하는데
코드 수정  →  수정 = [0.3, -0.1, 0.8, ...]   # refactor 해야 하는데
```

Attention은 이걸 해결한다. **단어가 주변 단어를 참조해서 자신의 벡터를 문맥에 맞게 업데이트**한다.

#### 추가 설명

06까지는 수정 이라는 단어 하나만 보고 답을 냈어.
```
수정 → fix   (마지막에 학습된 게 이기는 구조)
```

07 Attention부터는 수정 앞에 뭐가 있는지 같이 봐.

```
버그 + 수정  →  수정이 버그를 참조  →  fix
코드 + 수정  →  수정이 코드를 참조  →  refactor
```

같은 단어 수정인데 문맥에 따라 다른 벡터가 되는 것.

06단계(Embedding)까지는 단어를 **독립적으로** 처리했다.

## 2. Q, K, V란 무엇인가

```python
attn_out, attn_weights = self.attention(emb, emb, emb)  # Q(Query), K(Key), V(Value)
```

같은 `emb`를 3번 넣는다 (Self-Attention이기 때문).  
내부에서 각각 **다른 가중치 행렬**을 곱해 목적이 다른 벡터 3벌을 만든다.

```
emb (8차원)  →  W_Q 곱  →  Q (8차원)   "나는 뭘 찾고 있나?" (수정이 묻는다)
emb (8차원)  →  W_K 곱  →  K (8차원)   "나는 어떤 단어야?"  (버그, 코드 가 대답)
emb (8차원)  →  W_V 곱  →  V (8차원)   "참조 비중 결정되면 실제로 가져올 정보"
```

W_Q, W_K, W_V는 학습되는 가중치다. 즉 같은 입력인데 **다른 필터 3개를 통과**시켜서 목적이 다른 벡터 3개를 만드는 것.

문장이 2단어면 단어마다 Q, K, V가 생기므로:

```
['버그', '수정']
  버그  → Q(8) K(8) V(8)
  수정  → Q(8) K(8) V(8)

총 2단어 × 3벌 × 8차원이 동시에 존재
```

## 3. Attention 계산 흐름

### 케이스 1: `버그 수정` → `bug fix`

```
수정(Q) · 버그(K) → 점수 0.8   ← "수정은 버그를 많이 참조"
수정(Q) · 수정(K) → 점수 0.2   ← "수정은 자기 자신은 조금 참조"

softmax([0.8, 0.2]) → [0.69, 0.31]   ← 참조 비중 확정

수정의 새 벡터 = 0.69 × 버그(V) + 0.31 × 수정(V)
             = 버그 정보가 많이 섞인 8차원 벡터
             → fc 통과 → fix
```

### 케이스 2: `코드 수정` → `code refactor`

```
수정(Q) · 코드(K) → 점수 0.8   ← "수정은 코드를 많이 참조"
수정(Q) · 수정(K) → 점수 0.2   ← "수정은 자기 자신은 조금 참조"

softmax([0.8, 0.2]) → [0.69, 0.31]   ← 참조 비중 확정

수정의 새 벡터 = 0.69 × 코드(V) + 0.31 × 수정(V)
             = 코드 정보가 많이 섞인 8차원 벡터
             → fc 통과 → refactor
```

### 핵심

```
참조 강도는 두 케이스 모두 0.8로 같다
근데 섞이는 V가 달라 (버그(V) vs 코드(V))
그래서 수정의 최종 벡터가 달라지고 → 결론도 달라진다

버그 수정의 수정 벡터 ≠ 코드 수정의 수정 벡터
같은 단어지만 문맥에 따라 다른 벡터가 되는 것
```

결과적으로 `버그 수정`의 `수정`과 `코드 수정`의 `수정`은  
**같은 단어지만 다른 벡터**가 된다.

## 4. 출력은 새로운 8차원이다

### ❓ 이 결과물은 원래의 8차원인가, 새로 만든 8차원인가?

**새롭게 만들어진(업데이트된) 8차원**이다.  
원래 단어 벡터가 주변 문맥을 고려해 **"더 똑똑해진 8차원"** 으로 변환된 것.

#### 변환 4단계

```
1. 선형 변환  : emb × W_Q/K/V → Q, K, V (각각 8차원)
2. 쪼개기     : num_heads=2 → 앞 4차원(Head1), 뒤 4차원(Head2)
3. 독립 연산  : Head1, Head2 각자 4차원 안에서 Attention 계산
4. 병합       : Head1(4) + Head2(4) → concat → 새로운 8차원
```

## 5. 잔차 연결 (Residual Connection)

실제 Transformer에서는 기존 값을 덮어쓰지 않는다.

```
최종 단어 배열 = 기존 단어 배열 + Attention 결과 배열
```

Attention 결과를 굳이 입력과 같은 8차원으로 맞춰야 하는 이유가 여기 있다.  
**같은 인덱스끼리 1:1 덧셈**이 가능해야 하기 때문.

이를 통해 단어 본연의 의미에 문맥 정보가 **누적**된다.  
(07.py 예제 코드에는 잔차 연결이 생략되어 있음)

## 6. num_heads의 의미

### num_heads는 입출력 단어 수와 무관하다

`embed_dim=8`이면 가능한 헤드 수: **1, 2, 4, 8** (나눠떨어져야 함)  
07.py에서 `num_heads=2`인 건 입력 단어가 4개라서가 아니라,  
8차원을 나눌 수 있는 가장 단순한 값이기 때문.

### 헤드 수에 따른 차이

```
num_heads=1~2  (소규모)
  → 굵직한 관계 위주 학습
  → 단순한 규칙 (버그→fix, 코드→refactor) 에 충분

num_heads=8~16 (대규모, GPT 등)
  → 시제, 주동 관계, 감정선, 동음이의어 등 미세한 특징까지 분석
  → embed_dim도 512, 1024 이상으로 커야 함
```

head가 많다고 무조건 좋은 게 아니다.  
데이터가 단순한데 head가 많으면 오히려 과적합(Overfitting)된다.

## 7. 블랙박스 문제

Attention 가중치를 시각화하면 `수정`이 `버그`를 얼마나 참조했는지는 볼 수 있다.

```
버그 → 수정  참조 강도 0.8  →  fix
코드 → 수정  참조 강도 0.8  →  refactor
```

하지만 **왜 같은 참조 강도에서 다른 결론이 나오는지**는 알 수 없다.  
W_Q, W_K, W_V, fc 가중치들이 복합적으로 얽혀 결정하기 때문.

이게 AI 전체를 관통하는 블랙박스 문제다.

```
퍼셉트론  → 왜 학습되는지 완전히 설명 못함
역전파    → 왜 수렴하는지 수학적 보장 없음
딥러닝    → 레이어가 뭘 학습하는지 여전히 모름
Attention → 헤드가 뭘 배웠는지 추측만 가능
GPT       → 왜 추론이 되는지 아무도 몰라
```

단계마다 블랙박스가 해결된 게 아니라 **더 커진 것**이다.  
파라미터 수십 개 → 수천억 개, 모를수록 더 잘 되는 아이러니.

## 8. W_Q, W_K, W_V도 학습된다

`backward()` → `optimizer.step()` 할 때 fc 가중치만 업데이트되는 게 아니다.

```python
self.optimizer = optim.Adam(self.parameters(), ...)
#                                  ↑
#     embedding, W_Q, W_K, W_V, fc 전부 포함
```

`self.parameters()`가 모델 안의 학습 가능한 파라미터를 전부 묶어서 넘긴다.  
한 번 `step()` 호출에 전부 같이 움직인다.

```
학습 전
  버그(emb) × W_K → K [0.3, 0.1, ...]   # 랜덤 초기값
  코드(emb) × W_K → K [0.3, 0.1, ...]   # 거의 비슷함

학습 후
  버그(emb) × W_K → K [0.8, -0.2, ...]  # fix 방향으로 업데이트됨
  코드(emb) × W_K → K [0.1,  0.9, ...]  # refactor 방향으로 업데이트됨
```

---

## 9. Embedding과 W_Q/K/V의 차이

둘 다 `self.parameters()` 안에서 같이 업데이트되지만 **학습하는 대상이 다르다.**

```
embedding      →  단어 자체의 의미 벡터 학습
                  "버그" 라는 단어가 어떤 벡터인지

W_Q, W_K, W_V →  단어 간 관계 학습
                  "버그 옆에 수정이 오면 어떻게 참조할지"
```

```
학습 전
  embedding["버그"] = [0.1, 0.3, ...]   # 랜덤
  W_K               = [[0.2, ...], ...]  # 랜덤

학습 후
  embedding["버그"] = [0.8, -0.2, ...]  # "버그다움" 이 담김
  W_K               = [[0.9, ...], ...]  # "버그가 수정 옆에 있을 때" 가 담김
```

비유하면:
- `embedding` → 단어 사전 (단어 자체가 뭔지)
- `W_Q, W_K, W_V` → 문법책 (단어들이 서로 어떻게 관계 맺는지)

### Embedding이 중요한 이유

```
embedding  →  "나 버그야, 8차원 공간에서 여기 있어"  (위치/의미 확정)
W_Q, W_K   →  "버그가 수정이랑 얼마나 관련있어?"     (관계 확인)
```

```
embedding이 엉터리면
  → Q, K, V 계산 자체가 쓰레기
  → Attention이 아무리 정교해도 소용없음
```

**Garbage in, Garbage out**

그래서 실제 모델들은 embedding만 따로 사전학습 시키기도 한다.  
`Word2Vec`, `GloVe` 같은 게 그거다. embedding을 잘 만들어두고 그 위에 Attention을 얹는 구조.
